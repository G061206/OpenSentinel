你是一名资深全栈工程师与技术负责人，请为我从零开发一个“LLM 驱动的持续事件跟踪器”项目，并以**可在 Ubuntu VPS 上稳定部署**为首要目标。

## 一、项目目标

开发一个可以长期运行的事件跟踪系统。用户可以创建一个“跟踪任务”，例如：

* 美伊是否开战
* 某国是否宣布出口管制
* 某公司是否发布重大裁员
* 某地区是否发生重大地震/冲突升级

系统需要自动持续搜索公开信息源，定时更新，识别事件进展，生成结构化报告，并通过企业微信群 webhook 推送结果。

这不是一次性问答工具，而是一个**长期运行的监测系统**。

---

## 二、核心功能范围

### 1. 跟踪任务管理

系统必须支持：

* 新增跟踪任务
* 编辑跟踪任务
* 暂停/恢复跟踪任务
* 删除跟踪任务
* 查看任务列表
* 查看单个任务详情
* 手动触发一次立即执行

每个跟踪任务至少包含以下字段：

* `title`：任务名称
* `question`：要持续回答的问题
* `description`：背景说明
* `queries`：搜索关键词数组，可多语言
* `source_profile`：信源模板
* `schedule`：执行频率，例如每 15 分钟 / 每 1 小时 / 每天 9 点
* `alert_rules`：触发预警的规则
* `delivery_channels`：推送目标，例如企业微信群 webhook
* `status`：active / paused / archived
* `priority`
* `created_at / updated_at`

---

### 2. 采集与搜索

系统需要定时执行搜索任务，支持以下信源类型：

* RSS/Atom
* 普通网页抓取
* 可选接入新闻搜索 API
* 为后续扩展预留统一 adapter 接口

要求：

* 统一输出标准化的 `RawItem`
* 支持多来源并发采集
* 支持失败重试
* 支持超时控制
* 支持基础限流
* 支持去重

请优先做一个易扩展的 **source adapter 架构**。

---

### 3. 内容归一化与去重

对采集到的原始内容，需要做以下处理：

* 提取标题、正文、发布时间、来源、URL
* 清洗噪声内容
* 生成稳定哈希
* URL 去重
* 标题相似去重
* 内容近重复去重
* 把相似内容聚合到同一事件簇（cluster）中

需要设计合理的数据模型来支持：

* 原始条目
* 归一化条目
* 事件簇
* 证据列表

---

### 4. 事件判断与状态升级

系统不能只做“新闻摘要”，还要做“事件状态判断”。

请为每个跟踪任务设计一个可扩展的状态机，例如：

* `NORMAL`
* `WATCH`
* `ELEVATED`
* `CRISIS`
* `CONFIRMED`

状态切换不能完全依赖 LLM 幻觉，必须采用：

* **规则判断**
* **来源可信度**
* **数量阈值**
* **LLM 复核**

例如：

* 若高可信来源在一定时间窗内多次报道同一核心事实，则提升状态
* 若仅低质量来源出现一次，则保持观察状态
* 若事件定义满足任务问题中的“确认条件”，才进入 `CONFIRMED`

LLM 的输出必须尽量结构化，避免让模型直接返回自由发挥的大段结论。

---

### 5. 报告生成

系统需要生成两类内容：

#### A. 实时预警

当事件达到规则阈值时，生成简短告警内容，包括：

* 当前判断
* 关键事实
* 引用证据
* 来源列表
* 风险说明
* 时间戳

#### B. 周期性报告

例如每日上午生成摘要报告，包括：

* 当前状态
* 过去一段时间的主要新增进展
* 关键证据
* 信息空白点
* 下一步建议关注点

报告输出格式优先为：

* Markdown
* 结构化 JSON（内部）
* 便于转为企业微信 markdown 消息

## 三、技术约束

### 1. 部署环境

首要部署目标是 **Ubuntu VPS**，要求：

* 单机可运行
* Docker Compose 可部署
* 尽量减少外部依赖数量
* 配置简单
* 便于后续扩展为多服务

### 2. 推荐技术栈

优先按以下技术栈实现：

* **Python 3.11+**
* **FastAPI**：API 与后台服务
* **SQLAlchemy / SQLModel**
* **PostgreSQL**
* **pgvector**（如确有必要）
* **Redis**
* **Celery + Celery Beat** 或同等级方案
* **httpx**
* **feedparser**
* **trafilatura / readability**
* **Docker Compose**

2. 若你认为必要，再使用 React/Next.js，但必须说明理由

默认选择**更轻、更适合 VPS 的方案**，不要为了“现代化前端”显著增加复杂度。

## 五、LLM 使用原则

LLM 在本项目中主要用于：

* 信息归纳
* 事实抽取
* 证据结构化
* 事件状态复核
* 报告生成

不要把 LLM 用于：

* 直接代替采集
* 直接代替数据库逻辑
* 完全代替规则判断
* 无约束地产出最终状态结论

请优先设计以下模式：

1. **先规则初筛**
2. **再 LLM 结构化抽取**
3. **再规则与 LLM 结果融合**
4. **最后生成用户可读报告**

LLM 输出尽量使用 JSON schema 或严格字段约束。
